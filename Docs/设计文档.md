## 1. 总体架构设计

### 1.1 分层架构

系统采用 **分层架构 + MVVM 模式**，自上而下分为四层：

```
┌──────────────────────────────────────────────────────────┐
│                      表现层 (WPF)                          │
│  MainWindow  ·  LoginWindow  ·  FullscreenEnhanceWindow    │
│  MediaPlayerWindow  ·  ImageViewerWindow  ·  各 Dialog      │
│  (XAML 数据绑定 + IValueConverter + Command)                 │
├──────────────────────────────────────────────────────────┤
│                    ViewModel 层 (MVVM)                      │
│  MainViewModel  (partial class, 5 个功能文件)                │
│  LoginViewModel                                            │
│  CommunityToolkit.Mvvm 源码生成器                            │
├──────────────────────────────────────────────────────────┤
│                    业务逻辑层 (Services)                     │
│  DataService  ·  AuthService  ·  AiService                 │
│  FileScanService  ·  PlaybackService  ·  ThumbnailService   │
│  VideoEnhancer  ·  ScreenRecorder                          │
├──────────────────────────────────────────────────────────┤
│                    增强方法层 (Core)                         │
│  EnhancementRegistry (注册中心)                              │
│  LinearStretchMethod  ·  MultinexNanoMethod                  │
│  OnnxModelHelper (ONNX 预处理/后处理)                        │
├──────────────────────────────────────────────────────────┤
│                    数据访问层 (Data)                         │
│  AppDbContext (EF Core)  ·  Migrations                      │
│  SQLite 数据库 (MediaEnhancerDb.sqlite)                     │
└──────────────────────────────────────────────────────────┘
```

**层间依赖规则**：
- 上层可依赖下层，下层**不可**依赖上层
- ViewModel 通过 DI 注入 Service 接口，不直接依赖具体实现
- 增强方法层独立于业务逻辑层：`EnhancementRegistry` 是全局单例，被 `ScreenRecorder`、`VideoEnhancer`、`FullscreenEnhanceWindow` 消费
- 数据访问层的 `AppDbContext` 通过 `IDbContextFactory` 由各 Service 按需创建

### 1.2 MVVM 模式选型理由

| 考量维度 | 选型 | 原因 |
|----------|------|------|
| MVVM 框架 | CommunityToolkit.Mvvm 8.4 | 微软官方推荐，源码生成器消除样板代码 |
| 命令绑定 | `[RelayCommand]` | 自动生成 `ICommand` 属性，支持异步 |
| 属性通知 | `[ObservableProperty]` | 自动生成 `INotifyPropertyChanged` 代码 |
| DI 容器 | Microsoft.Extensions.DependencyInjection | .NET 官方 DI，与 EF Core 原生集成 |

**MainViewModel 采用 partial class 拆分为 5 个文件**：
- `MainViewModel.cs` — 文件管理、搜索筛选、批量操作
- `MainViewModel.Dashboard.cs` — 数据统计仪表盘
- `MainViewModel.Enhancement.cs` — 实时/离线增强、录屏
- `MainViewModel.Ai.cs` — AI 对话、AI 图像编辑
- `MainViewModel.Settings.cs` — 系统设置

> 拆分原因：单一文件会超过 2000 行，不利于阅读和维护。partial class 在编译后合并为一个类，不影响运行时行为。

### 1.3 依赖注入设计

所有服务的生命周期在 `App.xaml.cs` 中配置：

| 生命周期 | 服务 | 理由 |
|----------|------|------|
| Singleton | `EnhancementRegistry` | 全局唯一的增强方法注册中心 |
| Singleton | `IAuthService` / `AuthService` | 全局唯一的登录状态 |
| Singleton | `AiService` | 维护 HTTP 客户端连接池 |
| Scoped | `IDataService` | 按需获取 DbContext |
| Scoped | `IFileScanService`、`IPlaybackService`、`IThumbnailService` | 无共享状态 |
| Transient | `MainViewModel`、`LoginViewModel` | 每次解析获取新实例 |
| Transient | `MainWindow`、`LoginWindow` | 窗口每次重新创建 |

## 2. 增强插件架构设计

### 2.1 接口体系

增强方法层是整个系统**架构最精巧**的部分，定义了四层接口体系：

```
                   IEnhancementMethod (根接口)
                   ├─ Name: string
                   ├─ Description: string
                   └─ Parameters: List<EnhancementParameter>
                            │
          ┌─────────────────┼─────────────────┐
          │                 │                 │
  INativeEnhancement   IOnnxEnhancement   IRealTimeEnhancer
  (同步 C# 原生算法)    (异步 ONNX 推理)   (逐帧实时增强)
          │                 │                 │
          │  Enhance()       │  EnhanceAsync() │  Enhance(byte[])
          │  → BitmapSource  │  → Task<BS>     │  → byte[]
          │                 │                 │
          └─────────┬───────┘        SupportsRealTime
                    │                GetParameters()
          LinearStretchMethod   LinearStretchMethod
          (实现 INative)        (实现 IRealTime)
                                MultinexNanoMethod
                                (实现 IRealTime + IOnnx)
```

### 2.2 双路径执行设计

核心洞察：实时逐帧增强与离线文件增强有**完全不同的性能约束**，因此设计了两条独立执行路径：

**路径一：实时路径（byte[] → byte[]）**
- 面向 `FullscreenEnhanceWindow` 全屏覆盖和 `ScreenRecorder` 后处理
- 输入/输出均为 BGRA32 原始字节数组
- **零 WPF 对象分配**，避免 GC 触发导致掉帧
- `IRealTimeEnhancer.Enhance(byte[] pixels, int width, int height, int stride, ...)`
- 典型性能：线性拉伸 < 0.1ms/帧，Multinex Nano ~15-25ms/帧

**路径二：离线路径（BitmapSource → BitmapSource）**
- 面向文件增强（单图、视频逐帧）
- 使用 WPF `WriteableBitmap` 或 `BitmapImage`/`JpegBitmapEncoder`
- 支持异步 ONNX 模型推理
- `INativeEnhancement.Enhance(BitmapSource)` 或 `IOnnxEnhancement.EnhanceAsync(BitmapSource)`

### 2.3 插件注册与发现

`EnhancementRegistry` 是**注册表模式**实现，管理两套独立字典：

```csharp
// 运行时注册（App.xaml.cs OnStartup）
var registry = new EnhancementRegistry();
registry.Register(new LinearStretchMethod());      // 总是可用
try {
    registry.RegisterOffline(new MultinexNanoMethod());  // ONNX 注册，失败时静默跳过
} catch { /* ONNX Runtime 不可用时忽略 */ }

// 消费方通过接口获取
var method = registry.Current;                      // 当前选中的实时方法
var names = registry.RealTimeMethodNames;           // UI 下拉框绑定
var offline = registry.GetOfflineMethod("Multinex Nano");  // 离线增强用
```

**关键设计决策**：
- 第一个注册的方法**自动成为当前方法**（`Current` 属性），确保全屏增强按钮始终可用
- `IRealTimeEnhancer.SupportsRealTime` 属性控制该方法是否出现在实时增强方法列表中
- `MultinexNanoMethod` 的注册放在 `try-catch` 中——如果 ONNX Runtime DLL 缺失或模型文件损坏，系统仍可正常运行（只降级为线性拉伸）

### 2.4 可扩展性

新增一个增强方法只需三步：

1. 新建类，实现 `IRealTimeEnhancer` 和/或 `INativeEnhancement` 和/或 `IOnnxEnhancement`
2. 在 `App.xaml.cs` 中注册到 `EnhancementRegistry`
3. UI 自动发现该方法（`MethodNames` 属性绑定到下拉框）

**无需修改任何窗口、ViewModel 或 Service 代码**。

### 3.2 关键索引与约束设计

| 表 | 约束/索引 | 设计意图 |
|----|-----------|----------|
| `MediaFiles` | `(FilePath, UserId)` 复合唯一索引 | 同一用户不可重复导入相同路径的文件；不同用户可以导入同一路径 |
| `MediaFiles` | `SourceFileId` 自引用FK，`ON DELETE SET NULL` | 增强文件关联源文件；删除源文件时保留增强文件 |
| `Favorites` | `MediaFileId` 唯一索引 | 一个文件只能收藏一次（一对一） |
| `Recordings` | `MediaFileId` 唯一索引 | 一个录屏文件对应一条录制记录 |
| 所有子表 | `UserId` FK + 非唯一索引 | 按用户快速过滤，级联删除（删用户时清理所有数据） |
| `Users` | `Username` 唯一索引 | 防止重名注册 |

### 3.3 用户数据隔离方案

系统通过**两层隔离**确保多用户数据互不可见：

1. **应用层过滤**：`DataService` 的所有查询方法在构造时获取 `IAuthService.CurrentUser.Id`，每次访问数据库都在查询中附加 `.Where(x => x.UserId == currentUserId)`
2. **配置文件隔离**：每个用户的 API Key 和路径配置存储在独立文件 `appsettings_{userId}.json` 中，global 默认配置为 `appsettings.json`

### 3.4 级联删除策略

```
删除 User → 级联删除其所有 MediaFiles / PlayHistories / EnhancementLogs / Recordings / Favorites / RealtimeSessions
删除 MediaFile → 级联删除其 PlayHistories / EnhancementLogs / Recording / Favorite
删除 MediaFile（作为源文件）→ SET NULL 所有 EnhancedFile 的 SourceFileId
```

### 4.2 实时增强覆盖窗口的技术要点

全屏增强窗口（`FullscreenEnhanceWindow`）是整个系统**技术难度最高**的组件。三个核心问题及其解决方案：

**问题一：鼠标穿透**
- ❌ 错误方案：`IsHitTestVisible=False` — 虽然 WPF 不处理事件，但事件仍被窗口吞掉
- ✅ 正确方案：`SetWindowLong(hwnd, GWL_EXSTYLE, WS_EX_TRANSPARENT)` — 窗口对输入完全透明
- ⚠️ 必要条件：设置 `WS_EX_TRANSPARENT` 的窗口**必须是 Layered Window**（`AllowsTransparency=True`），因为顶层窗口中的非分层窗口没有可转发到的"下层窗口"

**问题二：递归捕获（增强窗口覆盖屏幕 → DXGI 捕获到自己 → 无限循环）**
- ✅ 解决方案：`SetWindowDisplayAffinity(hwnd, WDA_EXCLUDEFROMCAPTURE)`，Windows 10 2004+ 支持
- DXGI Desktop Duplication 在枚举帧时会自动跳过设置了该标记的窗口

**问题三：DPI 感知**
- 直接设置 `BitmapSource` 的 `DpiX/DpiY = 96`（而非系统 DPI），确保 1:1 像素映射
- 窗口覆盖所有屏幕（`VirtualScreen`），`WindowState=Normal`（非 Maximized，避免 DPI 缩放）

### 4.3 录屏后处理增强设计

```
录制阶段 (保证帧率)
  DXGI/GDI 捕获 → JPEG 编码 → 写入磁盘 (rec_{timestamp}_frames/)
  ⚡ 不做增强 —— 避免影响录制帧率

停止录制
  ├── 用户选择"增强录制"?
  │   ├── 是 → 逐帧读取 JPEG → IRealTimeEnhancer/IOnnxEnhancement → 覆盖写入
  │   │        (可中途取消 → 跳过增强，直接用原始帧编码)
  │   └── 否 → 使用原始帧
  ├── FFmpeg 编码: 帧序列 + 音频 → MP4
  │   编码器降级链: h264_nvenc → h264_qsv → h264_amf → libx264 → mpeg4
  └── 清理临时帧目录
```

## 5. AI 集成设计

### 5.1 API 抽象层

系统不绑定特定 AI 供应商，而是通过 **OpenAI 兼容 API 格式** 实现多供应商支持：

```
AiService
  ├── ChatAsync()      —— POST {endpoint}/chat/completions
  └── GenerateImageAsync()  —— DashScope multimodal-generation 或 OpenAI /images/generations

支持的服务商:
  ├── 阿里云 DashScope (通义千问 Qwen3-VL, 通义万相 Wan2.7)
  ├── DeepSeek
  ├── OpenAI 官方
  ├── 智谱 GLM
  └── Ollama 本地部署 (localhost)
```

### 5.2 模板降级回退

当用户未配置 API Key 或 API 调用失败时，系统自动**降级为本地模板分析**：

```
ChatAsync()
  ├── API 已配置? ──是──→ 构建多模态消息 → HTTP POST → 返回 AI 回复
  └── 否/异常 ─────→ FallbackChat()
                        ├── 检测 systemPrompt 关键词
                        ├── "简介" → 基于文件元数据生成结构化描述
                        ├── "数据" → 基于统计信息生成摘要
                        └── "增强" → 基于文件参数给出推荐
                        └── 附带提示: "当前未配置 AI API..."
```

这个设计的价值在于：**未配置 API 的用户仍能看到"AI 功能"的效果预览**，降低了功能入口的认知负担。

### 5.3 多模态支持

- **图片**：Base64 编码后嵌入消息的 `image_url` 字段
- **视频**：FFmpeg 抽取最多 3 个关键帧（I 帧），Base64 编码后嵌入
- **文件元数据**：类型、分辨率、时长、文件大小以文本形式附加到消息中

## 6. 安全设计

### 6.1 密码存储

```
注册:
  salt = RandomNumberGenerator.GetBytes(32)   // CSPRNG 随机盐
  passwordHash = SHA256(salt + password)       // 加盐哈希
  存储: PasswordHash (Base64) + Salt (Base64)

登录验证:
  hash = SHA256(存储的salt + 输入的password)
  比对: CryptographicOperations.FixedTimeEquals(hash, 存储的hash)
```

> 使用 `FixedTimeEquals` 而非 `==` 防止时序攻击（timing side-channel）。

### 6.2 API Key 加密

API Key 使用 **Windows DPAPI (Data Protection API)** 加密存储：

```
Protect(plaintext)
  → ProtectedData.Protect(Encoding.UTF8.GetBytes(plaintext), null, CurrentUser)
  → Base64 密文 → 写入 appsettings_{userId}.json

Unprotect(ciphertext)
  → Base64 解码 → ProtectedData.Unprotect(bytes, null, CurrentUser)
  → UTF8 明文
```

- 加密范围：`CurrentUser` — 只有当前 Windows 用户可解密
- 即使配置文件泄露，其他 Windows 用户或机器上的攻击者无法解密

参见 3.3 节。

## 7. 关键设计决策与权衡

### 决策一：为什么全屏增强窗口必须是 Layered Window？

| 方案 | 结果 |
|------|------|
| `WindowStyle=None, IsHitTestVisible=False` | ❌ 点击仍被窗口吞掉 |
| `WindowStyle=None + WS_EX_TRANSPARENT` | ✅ 真正的鼠标穿透 |
| `Popup` 控件 | ❌ 无法覆盖整个桌面 |

`WS_EX_TRANSPARENT` 要求窗口是 Layered Window（`WS_EX_LAYERED`），这就是必须设置 `AllowsTransparency=True` 的原因。

### 决策二：为什么录制时不实时增强？

| 指标 | 实时增强录制 | 后处理增强录制 |
|------|-------------|---------------|
| 录制帧率 | 可能掉帧（尤其是 ONNX 方法 15-25ms） | 始终稳定 |
| 增强质量 | 受限于实时方法 | 可用离线方法（质量更高） |
| 灵活性 | 无法中途取消增强 | 可随时取消增强 |
| CPU 占用 | 录制+增强同时 | 分阶段 |

选择后处理增强：**保证录制帧率是第一优先级**。

### 决策三：为什么选择 SQLite 而非 SQL Server / PostgreSQL？

| 考量 | SQLite | SQL Server |
|------|--------|------------|
| 部署复杂度 | 零配置，单文件 | 需安装实例 |
| 用户场景 | 桌面单机应用 | 多用户并发 |
| 打包体积 | < 2MB | > 200MB |
| 备份 | 复制 .sqlite 文件 | 需备份工具 |

作为单用户桌面应用，SQLite 是**天然最优解**。

### 决策四：为什么选择 EF Core 而非 Dapper / ADO.NET？

- EF Core 的 `Migrate()` 机制自动处理数据库升级
- LINQ 查询与 MVVM 数据绑定集成更自然
- 代码优先 (Code First) 开发效率高
- 对于本项目的查询复杂度，EF Core 性能完全足够

### 决策五：ENO8 页签为何全部定义在 MainWindow.xaml 中而非单独 UserControl？

- 8 个页面共享同一个 `MainViewModel`，无需创建子 ViewModel
- 页面间切换不需要重新加载，`Visibility` 切换即可
- 减少跨文件上下文切换的认知负担
- 代价：XAML 文件较长（约 2500 行），但不影响编译后性能

## 8. 设计模式应用

| 设计模式 | 应用位置 | 说明 |
|----------|----------|------|
| **策略模式** | `IEnhancementMethod` 接口体系 | 可运行时切换增强算法，客户端只依赖接口 |
| **注册表模式** | `EnhancementRegistry` | 集中管理所有增强方法，按名称查找 |
| **工厂模式** | `AppDbContextFactory` | 为 EF Core CLI 工具和 DI 提供 DbContext 实例 |
| **外观模式** | `DataService` | 将 EF Core 的 LINQ 操作封装为统一的数据访问接口 |
| **观察者模式** | `INotifyPropertyChanged` / `[ObservableProperty]` | View 自动响应 ViewModel 属性变化 |
| **命令模式** | `[RelayCommand]` / `ICommand` | 将 UI 操作封装为可绑定、可异步的命令对象 |
| **单例模式** | `EnhancementRegistry`、`AiService` | DI 容器管理，确保全局唯一实例 |
| **模板方法** | `VideoEnhancer.EnhanceAsync` | 固定四阶段管线（解帧→增强→合帧→编码），子步骤可替换 |
| **代理模式** | `AiService` | 封装外部 HTTP API 调用，提供缓存和降级逻辑 |

## 9. 并发与异步设计

### 9.1 异步模型

项目全面使用 `async/await` + `Task`：
- ViewModel 命令均为 `async Task`，UI 线程不阻塞
- 数据访问层所有操作返回 `Task<T>`，EF Core 原生异步支持
- 全屏增强窗口使用 `async void` 自调度循环（`Task.WhenAny` + `Task.Delay` 超时保护）
- 录屏使用 `CancellationTokenSource` 支持录制取消和增强阶段取消

### 9.2 线程安全

- **数据库操作**：`DataService` 通过 Scoped 生命周期确保每次操作获取独立 `DbContext`，避免并发冲突
- **UI 更新**：所有绑定通过 `INotifyPropertyChanged` 自动 marshall 到 UI 线程
- **ONNX Session**：`MultinexNanoMethod` 使用双重检查锁定（double-checked locking）保护 Session 的延迟初始化
- **帧缓存**：`FullscreenEnhanceWindow` 的帧处理使用单线程事件循环，天然的线程安全

## 10. 错误处理与容错设计

| 场景 | 处理策略 |
|------|----------|
| ONNX Runtime 不可用 | `try-catch` 静默跳过，系统降级为仅线性拉伸 |
| FFmpeg 未安装 | 仪表盘一键检查下载；各功能调用前检查，缺失时弹窗提示 |
| API Key 未配置 | AI 对话自动降级为模板分析 |
| 文件已被外部删除 | 文件校验功能检测，提示"定位文件"或"删除记录" |
| DXGI 捕获失败 | 自动回退到 GDI `CopyFromScreen` |
| 硬件编码器不可用 | 编码器降级链：NVENC → QSV → AMF → libx264 → mpeg4 |
| 中文路径兼容 | 视频处理时先复制到临时英文路径 |
| 密码/Key 解密失败 | `SecureStorage.Unprotect` 捕获异常，返回空字符串 |
