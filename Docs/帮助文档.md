## 一、图像增强方法详解

系统目前提供**两种增强方法**，各有不同的适用场景和性能特征。

### 1.1 线性拉伸

**原理**：通过统计图像中所有像素的亮度分布，将实际像素值范围线性映射到完整的 [0, 255] 动态范围。同时裁剪最暗和最亮各 2% 的像素以减少噪声干扰。

**工作流程**：
```
输入图像 → 计算亮度直方图 → 裁剪2%极值 → 线性映射到[0,255] → 输出
```

**参数**：

| 参数 | 范围 | 默认值 | 说明 |
|------|------|--------|------|
| 对比度 (Contrast) | 0.5 ~ 2.0 | 1.0 | 大于 1.0 增强对比，小于 1.0 减弱对比 |
| 亮度 (Brightness) | -50 ~ +50 | 0 | 正值提亮，负值压暗 |

**性能**：< 0.1ms/帧（1080p），纯 CPU 计算，无 GPU 依赖。

**适用场景**：低光照照片/视频、逆光拍摄、低对比度画面。适合实时增强。

### 1.2 Multinex Nano（ONNX 深度学习）

**原理**：基于 Retinex 理论的超轻量神经网络（仅 15K 参数），使用 ONNX Runtime 进行 CPU 推理。模型在 LOL-v2 合成数据集上训练，专为低光照增强设计。

**工作流程**：
```
输入图像 → 缩放至推理尺寸 → BGRA32→NCHW 张量 → ONNX 推理 → NCHW→BGRA32 → 缩放回原始尺寸 → 输出
```

**性能**：约 15-25ms/帧（1080p，CPU 推理）。可配置推理尺寸限制以提升速度。

**适用场景**：极低光照场景、需要更自然的增强效果。建议用于离线增强；在性能较好的设备上也可用于实时增强。

### 1.3 两种方法的对比

| 维度 | 线性拉伸 | Multinex Nano |
|------|----------|---------------|
| 处理速度 | < 0.1ms/帧 | 15-25ms/帧 |
| 增强效果 | 对比度提升明显，但可能过曝 | 更自然，保留细节 |
| GPU 需求 | 无 | 无（CPU 推理） |
| 可调参数 | 对比度 + 亮度 | 无（模型固定） |
| 文件大小 | 纯 C# 代码，< 200 行 | 依赖 ONNX Runtime + 模型文件 (~60KB) |
| 适用场景 | 实时增强、快速预览 | 离线增强、画质优先 |

## 二、全屏实时增强原理与故障排除

### 2.1 工作原理

全屏增强通过以下三步实现：

**第一步：屏幕捕获**
- 优先使用 **DXGI Desktop Duplication** API（DirectX 图形基础设施）
- 直接从 GPU 显存获取当前桌面纹理，**零 CPU 拷贝**
- 如 DXGI 初始化失败，自动回退到 **GDI CopyFromScreen**（较慢，作为保底方案）

**第二步：逐帧增强**
- 捕获的帧为 BGRA32 原始字节数组
- 调用当前选中的 `IRealTimeEnhancer.Enhance(byte[])` 方法
- **零 WPF 对象分配**（不创建任何 BitmapSource / WriteableBitmap），避免 GC 导致掉帧

**第三步：透明覆盖显示**
- 创建一个覆盖所有屏幕的无边框窗口
- 增强后的帧直接渲染到窗口中的 Image 控件
- 窗口设置了两个关键属性（见下文）

### 2.2 鼠标穿透机制

全屏增强窗口使用 `WS_EX_TRANSPARENT` 扩展窗口样式，使窗口对鼠标**完全透明**——点击事件会穿透到下方的应用程序，用户可以正常操作电脑。

> 简单的 `IsHitTestVisible=False` **不够**：WPF 不处理事件，但 Windows 消息循环仍然把事件发送到了我们的窗口，导致事件被吞掉。`WS_EX_TRANSPARENT` 从 OS 层面解决了这个问题。

### 2.3 防递归捕获

增强窗口覆盖在屏幕上 → DXGI 捕获屏幕 → 如果捕获到增强窗口本身 → 无限循环。

解决方法：调用 `SetWindowDisplayAffinity(hwnd, WDA_EXCLUDEFROMCAPTURE)`，告诉 DXGI Desktop Duplication 跳过此窗口。需要 Windows 10 2004 及以上版本。

### 2.4 故障排除

| 问题 | 可能原因 | 解决方法 |
|------|----------|----------|
| 全屏增强画面为黑色 | DXGI 捕获失败且 GDI 回退未生效 | 更新显卡驱动，重启程序 |
| 增强窗口挡住鼠标 | `WS_EX_TRANSPARENT` 未生效 | 确认 Windows 版本（需 Win10+），尝试以管理员身份运行 |
| 增强后画面闪烁 | 帧率不稳定或 DPI 缩放问题 | 尝试切换到线性拉伸方法，检查 Windows 显示缩放设置 |
| 按 F11 无法退出 | 全局热键注册冲突 | 切换回程序窗口点击"停止增强"按钮 |
| 增强画面与实际画面不同步 | 多显示器场景下捕获了错误屏幕 | 检查主显示器设置 |

## 三、屏幕录制增强流程详解

### 3.1 为什么要"后处理增强"？

屏幕录制采用**"先录制、后增强"**的分阶段策略：

```
【录制阶段】                       【后处理阶段】
目标是保证帧率稳定                 目标是用最好的质量增强

DXGI/GDI 捕获                     从磁盘读取 JPEG 帧
    ↓                                  ↓
JPEG 编码（极快）                   IRealTimeEnhancer.Enhance()
    ↓                                  ↓
写入磁盘                          覆盖保存增强帧
                                    ↓
                                  可选取消增强（用原始帧编码）
                                    ↓
                                  FFmpeg 编码为 MP4
```

**为什么不录制时实时增强？** 增强（尤其是 ONNX 推理 15-25ms）会显著降低录制帧率。将增强推迟到录制结束后，既保证了录制的流畅性，又可以使用更高画质的增强方法。

### 3.2 中途取消增强

录制停止后的后处理增强阶段，界面会显示"取消增强"按钮。点击后：
1. 立即停止逐帧增强
2. 直接使用已保存的原始帧进行 FFmpeg 编码
3. 生成不含增强效果的录制文件

> 适用场景：录制了一个很长的视频（如 1 小时），增强需要很长时间，想快速获得原始画面的视频文件。

## 四、视频增强处理流程

视频增强通过四个阶段完成：

```
阶段 1: FFmpeg 解帧
  输入视频.mp4  →  FFmpeg -q:v 2  →  temp/frame_00000001.jpg, frame_00000002.jpg, ...

阶段 2: 提取音频轨
  输入视频.mp4  →  FFmpeg -vn -acodec copy  →  temp/audio.aac

阶段 3: 逐帧增强
  foreach frame in temp/*.jpg:
    BitmapImage 加载 → 复制 BGRA32 像素 → IRealTimeEnhancer.Enhance(byte[]) → JpegBitmapEncoder 保存
  进度实时反馈到 UI

阶段 4: 合帧编码
  temp/frame_%08d.jpg + temp/audio.aac  →  FFmpeg 编码  →  输出.mp4
```

**耗时估算**：假设视频为 720p（1280×720），30 FPS，时长 10 秒，共 300 帧。使用线性拉伸（< 0.1ms/帧），阶段 3 约需 0.03 秒；使用 Multinex Nano（~20ms/帧），阶段 3 约需 6 秒。阶段 1 和阶段 4 的耗时取决于 FFmpeg 和磁盘 I/O。

## 五、AI 对话与编辑配置详解

### 5.1 API 端点（Endpoint）说明

系统使用 OpenAI 兼容的 Chat Completions API 格式，支持以下服务商：

| 服务商 | Chat Endpoint 示例 | 需要自行申请 |
|--------|-------------------|-------------|
| 阿里云 DashScope | `https://dashscope.aliyuncs.com/compatible-mode/v1` | 是（百炼平台） |
| OpenAI 官方 | `https://api.openai.com/v1` | 是 |
| DeepSeek | `https://api.deepseek.com/v1` | 是 |
| 智谱 AI (GLM) | `https://open.bigmodel.cn/api/paas/v4` | 是 |
| Ollama 本地 | `http://localhost:11434/v1` | 本地部署 |

### 5.2 API Key 的安全存储

你的 API Key 使用 **Windows DPAPI**（数据保护 API）加密存储：
- 加密后的 Key 以密文形式保存在 `appsettings_{userId}.json` 文件中
- 只有在**当前 Windows 用户账户**下才能解密
- 即使配置文件被复制到其他机器，也无法解密

### 5.3 图像生成格式说明

AI 编辑支持三种输出格式：

| 格式 | 说明 | 适用 Endpoint |
|------|------|--------------|
| Auto（推荐） | 自动检测 Endpoint，选择对应的调用格式 | 任意 |
| DashScope | 使用阿里云 DashScope 异步任务 API（提交 → 轮询 → 下载） | `dashscope.aliyuncs.com` |
| OpenAI | 使用 OpenAI 同步 `/images/generations` API | OpenAI 兼容 |

### 5.4 模型名称参考

| 用途 | 推荐模型 | 说明 |
|------|----------|------|
| AI 对话 | `qwen3-vl-flash`（DashScope） | 多模态，支持图像理解，性价比高 |
| AI 对话 | `deepseek-chat`（DeepSeek） | 纯文本对话，推理能力强 |
| AI 编辑 | `wan2.7-image`（DashScope） | 文生图/图生图 |

### 5.5 API 不可用时的降级行为

当 API 未配置或调用失败时，AI 对话功能自动使用**本地模板分析**。系统根据已选文件的基本元数据（文件名、类型、尺寸、大小等）生成结构化的文字说明。这确保了**即使没有网络和 API Key，基本功能仍然可用**。

## 六、编码器降级链说明

视频增强和录屏的最终编码阶段，系统按以下优先级自动尝试编码器：

| 优先级 | 编码器 | 类型 | 要求 |
|--------|--------|------|------|
| 1 | `h264_nvenc` | NVIDIA GPU 硬件编码 | NVIDIA 显卡 + 驱动 |
| 2 | `h264_qsv` | Intel GPU 硬件编码 | Intel 核显 + 驱动 |
| 3 | `h264_amf` | AMD GPU 硬件编码 | AMD 显卡 + 驱动 |
| 4 | `libx264` | CPU 软件编码 | 无特殊要求 |
| 5 | `mpeg4` | CPU 软件编码（兜底） | 无特殊要求 |

**降级逻辑**：系统按顺序尝试启动 FFmpeg 编码，如果当前编码器不可用（显卡不支持或 FFmpeg 未编译对应编码器），自动尝试下一个。最终保证在任意设备上都能成功编码。

**硬件编码速度参考**（相对于实际时长）：
- NVENC：约 2-5x 实时速度
- QSV：约 1.5-3x 实时速度
- libx264：约 0.5-2x 实时速度

## 七、数据安全与隐私说明

### 7.1 密码存储

- 用户密码使用 **SHA256 + 随机 Salt** 哈希存储
- 每次注册时生成 32 字节加密安全随机数作为 Salt
- 数据库中**不存储明文密码**
- 登录验证使用恒定时间比较（`FixedTimeEquals`），防止时序攻击

### 7.2 API Key 存储

- 使用 Windows DPAPI 加密，密文存储在本地 JSON 配置文件中
- 解密需要当前 Windows 用户身份
- 配置文件和数据库文件均在本地，**不上传到任何服务器**

### 7.3 AI 对话数据

- AI 对话内容通过 HTTPS 发送到配置的 API 端点
- 对话历史不持久化到数据库（ChatMessage 为纯内存模型）
- 发送给 AI 的图片和视频帧为实时 Base64 编码，不缓存在本地

### 7.4 多用户隔离

- 每个用户的数据通过 `UserId` 在数据库查询层完全隔离
- 每个用户的配置文件独立存储（`appsettings_{userId}.json`）
- 用户 A 无法看到用户 B 的文件、播放记录、增强历史、设置等

## 九、性能优化建议

### 9.1 实时增强性能

- **首选线性拉伸方法**：< 0.1ms/帧，几乎不消耗 CPU
- 使用 Multinex Nano 时注意：每帧约 15-25ms，在 60Hz 显示器上可能导致降帧。可通过降低捕获分辨率或在 ONNX Runtime 中配置推理线程数来优化
- 关闭不必要的后台程序，释放 CPU 资源

### 9.2 视频增强性能

- **分离输入和输出到不同磁盘**：将源视频放在 SSD A，输出路径设置在 SSD B，可减少 I/O 争抢
- 增强过程的帧处理是纯 CPU 操作，多核 CPU 表现更好
- 线性拉伸处理 4K 视频的时间约为 Multinex Nano 的 1/100

### 9.3 录屏性能

- **优先确保 DXGI 捕获生效**：检查 Windows 版本和显卡驱动
- 如果 GDI 回退启用，录屏帧率会较低（5-10 FPS），可以考虑降低录制分辨率
- 录制输出路径建议设在 SSD 上，减少写入延迟

### 9.4 数据库性能

- 文件数量超过 10,000 时，建议定期清理不再需要的文件和记录
- 缩略图缓存过期（> 30 天）会自动清理，也可以在数据统计页面手动触发
